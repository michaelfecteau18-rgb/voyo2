# VOYO / Adaptiride — Product Audit

**Author:** Product Owner agent
**Date:** 2026-06-23
**Scope:** Full repository + all product documentation
**Role boundary:** This audit is read-only. No source files were modified. Outputs are recommendations, backlog, and roadmap (see `tasks/backlog.md`, `docs/roadmap.md`).

---

## 1. Executive Summary

VOYO (repo name `adaptiride`, UI brand `movigo`) is an early-stage transportation-visibility platform for adapted/paratransit operators in Granby, Québec. The documented vision is strong and coherent: reduce phone calls, give dispatchers real-time visibility, and keep families informed. The current build delivers a genuinely working real-time GPS pipeline (driver → Firestore → dispatcher map), which is the hardest part of the vision and a real asset.

However, the project is **not currently demo-ready** and, as of this audit, **will not build**. Two defects block a production build, and several documented core features — most importantly the SMS/family communication pillar — are entirely mocked. The supporting "orchestration" framework (specs, tasks, workflows) is scaffolding with no content yet.

**Verdict:** Solid foundation and clear product thinking, undermined by two build-blockers and a large gap between what the dashboard *appears* to do (in mock data) and what it *actually* does. Closing the P0 items below would make a credible, honest demo achievable quickly.

**Demo-readiness today: 4/10.** Achievable: 8/10 after the P0 backlog is cleared.

## 2. Product & Business Alignment

The documentation is unusually clear for this stage and the build mostly follows it.

| Documented pillar | Status in build | Notes |
|---|---|---|
| **Visibility** | Partially real | Real driver GPS renders on the dispatcher map. Most other "visibility" (driver list, stops, ETAs) is mock data. |
| **Communication** | Mocked entirely | SMS log, "47 sent / 100% delivered", delay notifications are all hardcoded. No SMS provider is integrated. This is the single largest vision-vs-reality gap. |
| **Reliability** | Partial | Geofence arrival, wake-lock, GPS throttling, and trip-completion logic exist and are thoughtful — but trip completion is on the broken code path (see 4.1). |
| **Simplicity** | Strong | Driver flow (one route, big buttons, arrival overlay) is well-judged for the user. |

**Target market alignment (Amibus / adapted transport):** The driver workflow correctly centers accessibility context (wheelchair, boarding-assistance notes). This matches the buyer's actual operational need and is a good demo talking point.

**Audience risk:** `docs/users.md` and `product.md` name **Caregiver** and **Family Member** as primary users with notification needs, and `architecture.md` lists a Caregiver Portal and Family Portal as future apps. Neither exists in any form yet, and the notification mechanism they depend on (SMS) is mocked. The product's promised emotional payoff — "peace of mind" for families — is currently undeliverable.

## 3. What Is Real vs. Mocked

**Real / functional**
- Firebase email-password authentication (`login.jsx`).
- Driver live GPS: `watchPosition`, accuracy filtering, adaptive write throttling (60s stationary / 15s moving), screen wake-lock, writes to `trips/{id}` and `locations/{id}/points` (`ActiveRoute.jsx`).
- Geofence arrival detection (Haversine, 150 m threshold) with vibration + arrival overlay.
- Geocoding and turn-by-turn directions rendering via Google Maps.
- Dispatcher dashboard subscribes to active trips in real time (`onSnapshot`) and plots real GPS markers distinctly from mock drivers (`isReal` flag) — a genuinely honest design choice.
- Responsive admin layout (desktop sidebar / mobile tab bar).

**Mocked / hardcoded**
- All passengers (and they are inconsistent across the three files — see 5.2).
- All four dispatcher "drivers" (`MOCK_DRIVERS`).
- Entire SMS subsystem (`SMS_LOG`, counters).
- Activity feed (`MOCK_ACTIVITY`), passenger statuses, trip history.
- Driver home route summary ("4 passengers, 08h15 départ").

**Not built (empty scaffolding)**
- `docs/architecture.md` future portals; `docs/vision.md` (empty).
- `orchestration/orchestration.md`, `workflows.md` (empty).
- `specs/feature-driver-checkin.md.txt`, `specs/feature-parent-notifications.md.txt` (empty).
- `tasks/backlog.md`, `in-progress.md`, `completed.md` (empty — now seeded by this audit).
- `src/shared/components/`, `src/shared/hooks/` (empty directories).
- No tests anywhere (despite Builder/QA agents referencing them).

## 4. Critical Findings (Build-Blockers & Risks)

### 4.1 — P0 BUILD-BLOCKER: `await` in a non-async function
`src/driver/ActiveRoute.jsx`, `handleDropoff()` (line ~198) is declared `function handleDropoff()` but uses `await setDoc(...)` (line ~204). This is a parse-time error that will fail `vite build` and break the dev server on load. It sits on the **trip-completion path** — i.e., the most important moment of the driver demo (last passenger dropped off) is exactly what's broken.

### 4.2 — P0 BUILD-BLOCKER: case-mismatched imports
`src/App.jsx` imports `./shared/Login.jsx`, `./driver/DriverHome.jsx`, `./admin/AdminMap.jsx`, but the files on disk are `login.jsx`, `driverhome.jsx`, `adminmap.jsx`. (`ActiveRoute.jsx` is the only one that matches.) This works on case-insensitive Windows/macOS dev machines but **fails on case-sensitive Linux**, which is what CI and most hosting build steps use. The existing `dist/` was built before these mismatches and masks the problem.

### 4.3 — P0 SECURITY: no Firestore security rules in the repo
There is no `firestore.rules` file. Whatever rules exist live only in the Firebase console and are not version-controlled or reviewable. Given the app writes location data keyed to authenticated drivers, unmanaged rules are a real exposure (open read/write would let anyone tamper with trips/locations). Rules must be authored, committed, and deployed.

### 4.4 — P1 SECURITY: secrets committed to the repo
- Google Maps/Directions API key hardcoded in `ActiveRoute.jsx` and `adminmap.jsx`, and again in `Google API'S.txt`.
- Firebase config (incl. `apiKey`) in `firebase.js`.

The Firebase web `apiKey` is not strictly secret by design, but the **Maps key is** and is currently unrestricted in source and in a committed text file. It should be moved to env vars and restricted by HTTP referrer + API. `Google API'S.txt` should be removed from the repo and the key rotated.

### 4.5 — P1 SECURITY: routes are not auth-guarded
Only `DriverHome` checks `onAuthStateChanged`. `/driver/route` (`ActiveRoute`) and `/admin` (`AdminMap`) render with no auth check — they can be reached directly by URL. The admin/driver distinction is decided once at login by a hardcoded email string (`admin@movigo.ca`) and never enforced on the route, so any authenticated user can open `/admin`. Needs a shared route guard + role check.

### 4.6 — P2: GPS accuracy filter may suppress all updates
`if (accuracy > 50) return` silently drops every reading coarser than 50 m. Indoors, in vehicles, or on weak GPS this can mean the dispatcher map never updates and the geofence never fires — with no user-facing explanation. Needs a fallback/looser threshold and a "GPS weak" state.

## 5. Code Quality & Maintainability

### 5.1 — No shared data layer
Passengers, drivers, and routes are inlined as literals in components. There is no `src/data` or Firestore-backed passenger/route model, even though `architecture.md` defines Trips, Passengers, and Vehicles as core business objects with explicit state machines. The app cannot demo a second route or a real roster without code edits.

### 5.2 — Duplicated, inconsistent passenger data
The same passengers are redefined three times with conflicting fields: `driverhome.jsx` (status `pending`), `ActiveRoute.jsx` (full addresses, no status), `adminmap.jsx` (status `onboard`/`dropped`/`waiting`, SMS counts). A dispatcher and driver looking at "Marie Tremblay" see different truths. Single source of truth needed.

### 5.3 — Styling is 100% inline
Tailwind and PostCSS are configured but unused; every style is an inline object. This is verbose and blocks any design-system or theming work. Not urgent, but it's accruing debt against the documented "Mobile First / Human-Centered Design" principles.

### 5.4 — Bleeding-edge dependency versions
`package.json` pins React 19, Vite 8, ESLint 10, react-router 7. These should be verified as real, installed, and mutually compatible before a customer demo; very new majors are a common source of "works on my machine" build drift.

### 5.5 — Accessibility of the operator UI
Lots of low-opacity text (`rgba(255,255,255,0.3–0.45)`) on dark backgrounds likely fails WCAG AA contrast — relevant because drivers use this one-handed, in sunlight, in a moving vehicle. Worth a quick contrast pass before pilots.

## 9. What's Genuinely Good (preserve this)

- The real-time GPS pipeline is well-engineered (throttling, wake-lock, accuracy awareness, geofence + haptics).
- The dispatcher map honestly distinguishes real GPS from mock drivers — keep that integrity as more data goes live.
- Product docs are clear, user-centered, and correctly scoped ("What VOYO is not").
- Driver UX is appropriately simple for the operating context.

The foundation is real. The work ahead is mostly about closing the gap between the convincing mock dashboard and a smaller-but-true working system — and fixing two defects standing between the project and a build.
